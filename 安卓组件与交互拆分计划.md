# 安卓组件与交互拆分实施计划

> **供执行该计划的智能代理使用：** 必须使用 `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans` 按任务逐项实施。所有步骤使用复选框跟踪，任何阶段不得跳过失败测试、行为验证和提交边界。

**目标：** 在不全量重写原生 View、不改变现有用户行为的前提下，将 `FloatingChatOverlayUi.kt` 中混合的 UI、状态、交互和平台能力拆成边界清晰、可独立测试、可按需发布的 Android 组件。

**架构：** 采用混合 UI 架构。边缘手势、底部手势条、回退波浪和媒体表面继续使用原生 View；聊天、联系人、朋友圈、收藏、AI 与设置等状态密集业务继续使用 Compose。先在 `ubiki-accessibility` 内按功能拆包并稳定契约，满足复用门槛后再迁移到独立 Android Library Module。

**技术栈：** Kotlin 2.0.21、Jetpack Compose、Android View/Canvas、Android AccessibilityService、JUnit 4、Robolectric、Gradle Android Library、Macrobenchmark、JankStats、AAR、Maven Publish。

## 1. 决策摘要

### 1.1 接受的原则

- 组件输入统一为不可变 `Props` 或 `UiState`，输出统一为密封 `Event`。
- 业务组件不得直接依赖 `AccessibilityService`、`WindowManager`、数据库、HTTP 客户端或 Activity。
- Android 平台能力通过窄接口注入，由运行时适配层实现。
- 原生 View 和 Compose 都可以作为 Android 组件，不以是否继承 `View` 判断组件化程度。
- 组件先形成内部稳定边界，再决定是否成为独立 Module 或公共制品。
- 所有迁移以行为等价测试和真机性能数据为准，不以代码行数或框架偏好判断成败。

### 1.2 明确拒绝的做法

- 不把 2.3 万行 Compose 一次性改写成 `ViewGroup`。
- 不为每个按钮建立跨平台 JSON Schema。
- 不在没有 iOS 业务消费者时建设自制跨平台渲染引擎。
- 不把 `Modifier`、`Context`、`Uri`、Android `Color`、dp、sp 放入平台无关契约。
- 不因“可以发布”就立即把所有内部组件发布到 Maven。
- 不在缺少 Macrobenchmark/Perfetto 证据时声称 Compose 或 View 是性能根因。

### 1.3 最终技术边界

```text
业务层
  ├─ Feature UiState / Props
  ├─ Feature Event
  └─ Reducer / Coordinator
          ↓
Android Compose UI
  ├─ chat
  ├─ contacts
  ├─ moments
  ├─ media
  └─ tools
          ↓
Android Runtime Adapters
  ├─ AccessibilityService
  ├─ WindowManager
  ├─ Activity / Permission
  ├─ SQLite / SCRM
  └─ Media Surface

原生 View 独立路径
  ├─ EdgeOverlayView
  ├─ BottomGestureBarView
  ├─ BackWaveView
  └─ TextureView / SurfaceView
```

## 4. 接口契约规则

### 4.1 Props、State 与 Event

每个 Feature 使用平台无关 Kotlin 类型表达输入和输出：

```kotlin
data class ChatUiState(
    val conversation: FloatingChatConversation,
    val selectedThreadId: String?,
    val inputText: String,
    val selectedMessageIds: Set<String>,
    val loading: Boolean,
    val errorMessage: String?
)

sealed interface ChatUiEvent {
    data class InputChanged(val value: String) : ChatUiEvent
    data object SendClicked : ChatUiEvent
    data class MessageClicked(val messageId: String) : ChatUiEvent
    data class MessageLongPressed(val messageId: String) : ChatUiEvent
    data class ThreadSelected(val threadId: String) : ChatUiEvent
    data object CollapseRequested : ChatUiEvent
}
```

契约层禁止导入以下包：

```text
android.*
androidx.compose.*
java.awt.*
```

平台对象使用稳定标识替代，例如使用 `mediaId: String`，不把 `Uri` 放入契约。

### 4.2 一次性效果

导航、权限、系统选择器等不能作为持久状态，统一建模为 Effect：

```kotlin
sealed interface FloatingChatEffect {
    data class OpenDocument(val messageId: String) : FloatingChatEffect
    data class RequestPermission(val permission: PermissionKind) : FloatingChatEffect
    data class OpenMediaPicker(val kind: MediaPickerKind) : FloatingChatEffect
    data class ShowMessage(val text: String) : FloatingChatEffect
}
```

UI 只发送 Event；Coordinator 决定是否产生 Effect；Android Runtime Adapter 执行 Effect。

### 4.3 样式契约

只统一语义 Token，不统一平台布局对象：

```kotlin
data class FloatingChatSemanticStyle(
    val density: ContentDensity,
    val emphasis: Emphasis,
    val status: StatusTone
)

enum class ContentDensity { Compact, Standard }
enum class Emphasis { Low, Normal, High }
enum class StatusTone { Neutral, Info, Success, Warning, Error }
```

Android 端在 `FloatingChatTheme.kt` 中把语义映射为 Compose `Color`、`Dp` 和 `TextStyle`。未来 iOS 端可独立映射为 `UIColor`、point 和 `UIFont`，无需共享渲染实现。

## 5. 交互归属矩阵

| 交互 | UI 发送事件 | 状态处理者 | 平台执行者 |
|---|---|---|---|
| 输入文字 | `InputChanged` | Chat Coordinator | 无 |
| 发送消息 | `SendClicked` | Chat Coordinator | SCRM Port |
| 打开文档 | `DocumentClicked` | Media Coordinator | Activity Port |
| 选择图片 | `GalleryClicked` | Tool Coordinator | Media Picker Port |
| 相机拍摄 | `CameraClicked` | Tool Coordinator | Camera Activity Port |
| 请求录音权限 | `VoiceClicked` | Tool Coordinator | Permission Port |
| 切换账号 | `AccountSelected` | Shell Coordinator | Conversation Port |
| 联系人搜索 | `ContactQueryChanged` | Contacts Coordinator | SCRM Contact Port |
| 好友申请处理 | `FriendRequestAction` | Contacts Coordinator | SCRM Task Port |
| 点赞/评论朋友圈 | `MomentAction` | Moments Coordinator | Moment Port |
| 边缘手势 | 原生 `MotionEvent` | Gesture Detector | Action Executor |
| 收起悬浮窗 | `CollapseRequested` | Shell Coordinator | Window Port |

原则：Composable 不直接调用数据库、HTTP、Activity 或 Service；原生 View 不感知聊天业务。

## 7. 测试策略

### 7.1 单元测试

- Reducer：同一输入必须产生确定的新状态，不修改旧对象。
- Coordinator：事件只能调用所属 Port，不跨 Feature 产生副作用。
- Contract：禁止 Android 和 Compose 类型进入平台无关契约。
- SCRM：发送、轮询、失败和重试状态必须保持现有语义。
- 数据库：拆分 UI 不得改变 SQLite v5 表结构和迁移行为。

### 7.2 Compose UI 测试

- 使用稳定 `testTag`，不使用中文文案作为唯一定位方式。
- 覆盖默认、加载、空、失败、禁用和选择状态。
- 验证事件次数，避免一次点击触发重复发送。
- 验证不可见 Feature 不消费点击和手势。

### 7.3 原生 View 测试

- `EdgeOverlayView`、`BottomGestureBarView`、`BackWaveView` 保留现有实现。
- 覆盖 `ACTION_DOWN/MOVE/UP/CANCEL`。
- 验证 View 移除时取消回调，不持有失效 Service/Context。
- 真机验证系统返回手势、横屏、键盘和 Overlay 层级。

### 7.4 性能测试

- Macrobenchmark 测端到端启动、展开和滚动。
- JankStats 记录真实用户路径卡顿。
- Perfetto 仅用于定位已复现的慢帧，不凭静态代码猜测根因。
- Compose Layout Inspector 用于确认重组范围，但重组次数本身不是失败标准。

## 8. 风险与回滚策略

| 风险 | 控制措施 | 回滚点 |
|---|---|---|
| 一次移动过多代码导致行为漂移 | 每个 Feature 独立迁移、独立测试、独立提交 | 回退单个 Feature 提交 |
| Props/Event 变成万能对象 | 每个 Feature 自己定义契约 | 删除未被两个组件使用的字段 |
| Coordinator 演变成新巨型类 | 按 Chat/Contacts/Moments/Media/Tools 分 Coordinator | 保留 Shell 只做导航和组装 |
| Compose 重组变多 | 稳定 State、缩小状态读取范围、基准复测 | 恢复上一阶段组件边界 |
| View/Compose 互操作泄漏资源 | `DisposableEffect` 对称释放 | 恢复现有 Native Surface 路径 |
| 过早发布 API | 设置第二消费者和稳定迭代门槛 | 保留内部包，不发布制品 |
| 工作树已有未提交改动 | 每阶段先检查 `git diff`，不得覆盖用户修改 | 只提交本阶段明确文件 |
